# 1. Summary

This page explains how to **implement an MCP (Model Context Protocol) client** that connects an application to an MCP server.

### Key Concepts

* **Client role**: Acts as a bridge between application logic and MCP server.
* **Architecture**
  * `MCP Client`: Custom wrapper class
  * `Client Session`: Actual connection (from MCP SDK)
* **Why wrapper class**
  * Handles lifecycle (e.g., resource cleanup)
  * Simplifies usage for application logic

### Core Responsibilities

The client must:

* Retrieve available tools → `list_tools()`
* Execute tools requested by Claude → `call_tool()`

### Key Methods

```python
async def list_tools(self) -> list[types.Tool]:
    result = await self.session().list_tools()
    return result.tools
```

```python
async def call_tool(self, tool_name: str, tool_input: dict):
    return await self.session().call_tool(tool_name, tool_input)
```

### Application Flow

1. Application gets tools from MCP server
2. Sends tools + user query to Claude
3. Claude decides to call a tool
4. Client executes tool
5. Result sent back to Claude → final answer

👉 The client abstracts connection complexity and enables **tool-driven workflows**

# 2. Q\&A

### Q1. What is the purpose of an MCP client?

* To connect the application with the MCP server and expose its tools for usage.

### Q2. Why create a custom MCP client class?

* To manage session lifecycle and simplify interaction with the server.

### Q3. What are the two main components of the client?

* MCP Client (custom wrapper)
* Client Session (actual connection)

### Q4. What does `list_tools()` do?

* Retrieves all available tools from the MCP server.

### Q5. What does `call_tool()` do?

* Executes a specific tool with input parameters.

### Q6. How does Claude interact with tools?

* Claude selects tools → application executes via client → result returned.

### Q7. Why is resource cleanup important?

* Ensures sessions are properly closed to avoid leaks.

# 4. Key Takeaways

* MCP client is a **bridge layer** between application and MCP server
* Core operations are:
  * `list_tools()` (discovery)
  * `call_tool()` (execution)
* Custom wrapper class improves:
  * Maintainability
  * Resource handling
* Enables **Claude-driven tool execution loop**
* Fits into **agent-style workflows** (very relevant to LangChain / LangGraph mapping)
