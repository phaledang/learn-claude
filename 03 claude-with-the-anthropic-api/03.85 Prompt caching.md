### Problem (without caching)

* Each request:
  * Reprocesses identical content
  * Discards all intermediate computation after response
* Leads to:
  * Higher latency
  * Higher cost ### Solution (with caching)

* First request:
  * Performs full preprocessing and **stores results in cache**
* Subsequent requests:
  * **Reuse cached computation** instead of recomputing ## 2. Q\&A

**Q1. What is prompt caching?**  
Reusing preprocessing computation (tokens, embeddings, context) across repeated requests.

**Q2. What happens without prompt caching?**  
All preprocessing work is recomputed and discarded after each request.

**Q3. What is stored in the cache?**  
Internal representations such as tokenization and embeddings.

**Q4. When is caching most effective?**  
When the same large input appears repeatedly (e.g., document analysis).

**Q5. What are the main benefits?**  
Faster responses and lower cost.

**Q6. What is a key limitation?**  
Cache expires after about one hour.

## 4. Key Takeaways

* Prompt caching optimizes **compute reuse**, not output reuse
* Works best for **stable context + multiple queries**
* Directly impacts:
  * **Latency (↓)**
  * **Cost (↓)**
* Aligns strongly with:
  * RAG pipelines
  * Agent workflows with repeated context

## 5. Practical Architecture Insight (aligned with your GenAI work)

For solution design (e.g., Azure + Claude / hybrid systems):

```
User Query
   ↓
Shared Context (Document / Knowledge Base)
   ↓
[Prompt Cache Layer]
   ↓
LLM (Claude / Azure OpenAI)
```

### Optimization pattern

* Cache:
  * System prompt
  * Retrieved documents
  * Static instructions
* Vary:
  * User query
  * Task instructions

This pattern integrates well with:

* Multi-index RAG pipelines
* LangChain retrievers
* Agent loops (LangGraph)
