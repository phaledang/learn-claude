# 🧩 1. Tool Definition Syntax (API Request)

### Minimal required tool stub:

```json
{
  "type": "text_editor_20250728",
  "name": "str_replace_based_edit_tool"
}
```

### Optional parameter:

```json
{
  "max_characters": 10000
}
```

### Full usage in request:

```json
{
  "model": "claude-opus-4-8",
  "max_tokens": 1024,
  "tools": [
    {
      "type": "text_editor_20250728",
      "name": "str_replace_based_edit_tool",
      "max_characters": 10000
    }
  ],
  "messages": [
    {
      "role": "user",
      "content": "Fix the syntax error in primes.py"
    }
  ]
}
```

✅ Key points:

* `type` = versioned tool identifier
* `name` = tool name referenced in `tool_use`
* No `input_schema` required → predefined by Claude [\[platform.claude.com\]](https://platform.claude.com/docs/en/agents-and-tools/tool-use/text-editor-tool)

***

# 🧠 2. Tool Call Syntax (Claude → Application)

When Claude decides to use the tool:

### Tool call block structure:

```json
{
  "type": "tool_use",
  "id": "...",
  "name": "str_replace_based_edit_tool",
  "input": {
    "command": "view | str_replace | create | insert | undo_edit",
    ...
  }
}
```

✅ Observed pattern:

* `type: "tool_use"` → indicates tool invocation
* `name` → tool identifier from definition
* `input` → operation parameters

Claude emits this, application must execute it [\[platform.claude.com\]](https://platform.claude.com/docs/en/agents-and-tools/tool-use/overview)

***

# ⚙️ 3. Command-Level Syntax (Core Operations)

The text editor tool exposes a **fixed command vocabulary**:

### Command: `view`

```json
{
  "command": "view",
  "path": "./file.py",
  "view_range": [1, 100]
}
```

***

### Command: `str_replace`

```json
{
  "command": "str_replace",
  "path": "./file.py",
  "old_str": "foo",
  "new_str": "bar"
}
```

***

### Command: `create`

```json
{
  "command": "create",
  "path": "./new_file.py",
  "file_text": "print('hello')"
}
```

***

### Command: `insert`

```json
{
  "command": "insert",
  "path": "./file.py",
  "insert_line": 10,
  "new_str": "print('debug')"
}
```

***

### Command: `undo_edit`

```json
{
  "command": "undo_edit",
  "path": "./file.py"
}
```

✅ These commands are part of the **built-in schema**, not user-defined [\[simonwillison.net\]](https://simonwillison.net/2025/Mar/13/anthropic-api-text-editor-tool/)

***

# 🔄 4. Tool Result Syntax (Application → Claude)

After executing tool:

```json
{
  "type": "tool_result",
  "tool_use_id": "...",
  "content": "file contents or operation result"
}
```

✅ Required:

* must reference `tool_use_id`
* returned inside next message turn [\[platform.claude.com\]](https://platform.claude.com/docs/en/agents-and-tools/tool-use/text-editor-tool)

***

# 🔁 5. Full Execution Loop (Strict Syntax Flow)

```
1. API request
   → includes tools[]

2. Claude response
   → tool_use block

3. Application executes tool
   → returns tool_result

4. Claude continues reasoning
   → may call tool again or respond
```

✅ This loop is explicitly required for **client-side tools** [\[platform.claude.com\]](https://platform.claude.com/docs/en/agents-and-tools/tool-use/overview)

# 🔑 Key Takeaways (Syntax Perspective)

* Text editor tool = **predefined schema + custom execution**
* Only a **minimal stub** is needed in API request
* Claude generates:
  ```
  tool_use → JSON command
  ```
* System must return:
  ```
  tool_result → JSON response
  ```
* Command vocabulary is **fixed and structured**
* Entire interaction is a **strict JSON protocol loop**
