# 3) Raw Python + Claude — suggested RAG pipeline

## 3.1 Design note

Anthropic documentation explicitly documents the **Messages API** shape (`client.messages.create`) and the Python SDK usage. It does **not** document a built-in RAG framework in the sources gathered here, so the following is a **suggested application-side implementation** that uses Claude only for the final answer generation step. [\[platform.claude.com\]](https://platform.claude.com/docs/en/home), [\[github.com\]](https://github.com/anthropics/anthropic-sdk-python)

## 3.2 Suggested code

```python
import os
import math
import re
from typing import List, Tuple
from anthropic import Anthropic

# -----------------------------
# 1) Load source text
# -----------------------------
def load_text(path: str) -> str:
    with open(path, "r", encoding="utf-8") as f:
        return f.read()

# -----------------------------
# 2) Chunking
#    Suggested: size-based + overlap
# -----------------------------
def chunk_by_size(text: str, chunk_size: int = 300, overlap: int = 50) -> Listchunks = []
    start = 0

    while start < len(text):
        end = min(start + chunk_size, len(text))
        chunks.append(text[start:end])

        if end == len(text):
            break

        start = end - overlap

    return chunks

# -----------------------------
# 3) Embedding model adapter
#    Replace with real embedding provider
# -----------------------------
def embed_text(text: str) -> List"""
    Placeholder adapter.
    Plug in your embedding API here.
    """
    raise NotImplementedError("Connect your embedding model here.")

# -----------------------------
# 4) In-memory vector store
# -----------------------------
class InMemoryVectorStore:
    def __init__(self):
        self.rows: List[Tuple[str, List[float]]] = []

    def add(self, text: str, embedding: List[float]):
        self.rows.append((text, embedding))

    def cosine_similarity(self, a: List[float], b: List[float]) -> float:
        dot = sum(x * y for x, y in zip(a, b))
        norm_a = math.sqrt(sum(x * x for x in a))
        norm_b = math.sqrt(sum(x * x for x in b))
        return dot / (norm_a * norm_b + 1e-9)

    def search(self, query_embedding: List[float], top_k: int = 3) -> Listscored = []
        for text, emb in self.rows:
            score = self.cosine_similarity(query_embedding, emb)
            scored.append((score, text))

        scored.sort(reverse=True, key=lambda x: x[0])
        return [text for score, text in scored[:top_k]]

# -----------------------------
# 5) Build index
# -----------------------------
def build_index(path: str) -> InMemoryVectorStore:
    text = load_text(path)
    chunks = chunk_by_size(text)

    store = InMemoryVectorStore()
    for chunk in chunks:
        emb = embed_text(chunk)
        store.add(chunk, emb)

    return store

# -----------------------------
# 6) Query pipeline
# -----------------------------
client = Anthropic(api_key=os.environ["ANTHROPIC_API_KEY"])

def answer_question(store: InMemoryVectorStore, question: str) -> str:
    q_emb = embed_text(question)
    retrieved_chunks = store.search(q_emb, top_k=3)

    context = "\n\n---\n\n".join(retrieved_chunks)

    message = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=800,
        messages=[
            {
                "role": "user",
                "content": (
                    "Answer the question using only the retrieved context.\n\n"
                    f"Question: {question}\n\n"
                    f"Context:\n{context}"
                ),
            }
        ],
    )

    return message.content[0].text

# -----------------------------
# Example usage
# -----------------------------
if __name__ == "__main__":
    store = build_index("sample_input.txt")
    question = "What caused the payment service incident?"
    answer = answer_question(store, question)
    print(answer)
```

# 5) LangGraph — suggested end-to-end RAG workflow

## 5.1 Design note

LangGraph documentation defines `StateGraph` as a graph where nodes read and write shared state, and it explicitly requires `.compile()` before execution. LangGraph documentation also describes it as a low-level orchestration framework for long-running, stateful workflows. [\[reference....gchain.com\]](https://reference.langchain.com/python/langgraph/graph/state/StateGraph), [\[langchain-....github.io\]](https://langchain-ai.github.io/langgraphjs/reference/classes/langgraph.StateGraph.html)

This makes it a good fit when the RAG pipeline needs:

* explicit workflow steps,
* branching,
* retries,
* human-in-the-loop,
* or durable state.

That general positioning is explicitly stated in the docs. [\[reference....gchain.com\]](https://reference.langchain.com/python/langgraph/graph/state/StateGraph)

## 5.2 Suggested code

```python
from typing import TypedDict, List
from langgraph.graph import StateGraph, START, END
from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_anthropic import ChatAnthropic

class RAGState(TypedDict):
    text: str
    question: str
    chunks: List[str]
    retrieved_chunks: List[str]
    final_answer: str

def chunk_node(state: RAGState):
    splitter = RecursiveCharacterTextSplitter(
        chunk_size=300,
        chunk_overlap=50,
        length_function=len,
        is_separator_regex=False,
    )
    chunks = splitter.split_text(state["text"])
    return {"chunks": chunks}

def retrieve_node(state: RAGState):
    # Suggested placeholder retrieval logic.
    # Replace with a real vector store query.
    retrieved = state["chunks"][:3]
    return {"retrieved_chunks": retrieved}

def answer_node(state: RAGState):
    llm = ChatAnthropic(
        model="claude-sonnet-4-6",
        temperature=0,
    )

    context = "\n\n---\n\n".join(state["retrieved_chunks"])

    result = llm.invoke(
        f"Answer using only this context.\n\n"
        f"Question: {state['question']}\n\n"
        f"Context:\n{context}"
    )

    return {"final_answer": str(result)}

graph = StateGraph(RAGState)
graph.add_node("chunk", chunk_node)
graph.add_node("retrieve", retrieve_node)
graph.add_node("answer", answer_node)

graph.add_edge(START, "chunk")
graph.add_edge("chunk", "retrieve")
graph.add_edge("retrieve", "answer")
graph.add_edge("answer", END)

app = graph.compile()

with open("sample_input.txt", "r", encoding="utf-8") as f:
    text = f.read()

result = app.invoke(
    {
        "text": text,
        "question": "What caused the payment service incident?",
        "chunks": [],
        "retrieved_chunks": [],
        "final_answer": "",
    }
)

print(result["final_answer"])
```

## 5.3 Syntax explanation

* `StateGraph(RAGState)` defines a stateful workflow over a typed shared state. [\[langchain-....github.io\]](https://langchain-ai.github.io/langgraphjs/reference/classes/langgraph.StateGraph.html)
* `add_node(...)` adds workflow stages. [\[langchain-....github.io\]](https://langchain-ai.github.io/langgraphjs/reference/classes/langgraph.StateGraph.html)
* `add_edge(...)` defines execution order between nodes. [\[langchain-....github.io\]](https://langchain-ai.github.io/langgraphjs/reference/classes/langgraph.StateGraph.html)
* `compile()` is required before `invoke()`. [\[langchain-....github.io\]](https://langchain-ai.github.io/langgraphjs/reference/classes/langgraph.StateGraph.html)
* `invoke(...)` executes the compiled graph. The docs explicitly mention `invoke()` on compiled graphs. [\[langchain-....github.io\]](https://langchain-ai.github.io/langgraphjs/reference/classes/langgraph.StateGraph.html)

# 6) How to swap chunking strategies inside the same pipeline

The page currently open describes four strategies:

* **Size-Based**
* **Structure-Based**
* **Semantic-Based**
* **Sentence-Based** [\[bing.com\]](https://bing.com/search?q=LangGraph+StateGraph+official+docs)

A good pattern is to keep the rest of the RAG flow the same and only replace the **chunking stage**.

## Suggested swap points

### Size-Based

Use:

* raw Python `chunk_by_size(...)`
* LangChain `RecursiveCharacterTextSplitter(...)`
* LangGraph node wrapping either of the above

This aligns directly with the course page’s size-based strategy. [\[bing.com\]](https://bing.com/search?q=LangGraph+StateGraph+official+docs), [\[lagnchain....thedocs.io\]](https://lagnchain.readthedocs.io/en/stable/modules/indexes/text_splitters/examples/recursive_text_splitter.html)

### Structure-Based

Use:

* raw regex split on `##`
* LangChain `MarkdownHeaderTextSplitter(...)`
* LangGraph structure-aware chunk node

This aligns directly with the course page’s structure-based strategy and LangChain’s Markdown splitter docs. [\[bing.com\]](https://bing.com/search?q=LangGraph+StateGraph+official+docs), [\[lagnchain....thedocs.io\]](https://lagnchain.readthedocs.io/en/stable/modules/indexes/text_splitters/examples/markdown.html), [\[github.com\]](https://github.com/langchain-ai/langchain/blob/master/libs/text-splitters/langchain_text_splitters/markdown.py)

### Semantic-Based

Use:

* application-side embedding comparison in raw Python
* LangChain `SemanticChunker` in the experimental ecosystem docs
* LangGraph semantic chunk node

The course page explicitly describes semantic chunking as the most sophisticated and more computationally expensive strategy. [\[bing.com\]](https://bing.com/search?q=LangGraph+StateGraph+official+docs), [\[langchain-...thedocs.io\]](https://langchain-doc.readthedocs.io/en/latest/index.html)

### Sentence-Based

Use:

* regex sentence grouping in raw Python
* LangChain sentence-oriented splitters such as `NLTKTextSplitter`
* LangGraph sentence chunk node

This matches the page’s sentence-based middle-ground approach. [\[bing.com\]](https://bing.com/search?q=LangGraph+StateGraph+official+docs), [\[reference....gchain.com\]](https://reference.langchain.com/python/langchain-text-splitters)
