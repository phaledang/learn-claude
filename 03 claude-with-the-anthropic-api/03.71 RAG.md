# 📘 1. RAG

**Source:** [Introducing Retrieval Augmented Generation](https://anthropic.skilljar.com/claude-with-the-anthropic-api/287763)

## What is RAG?

* **Retrieval Augmented Generation (RAG)** is a technique for handling **large documents** that exceed prompt limits. [\[Introducin...Generation\]](https://anthropic.skilljar.com/claude-with-the-anthropic-api/287763)
* Instead of sending everything to the model, RAG:
  * Splits documents into **chunks**
  * Retrieves only the **most relevant chunks**
  * Sends those to the model for answering

## The Problem

* Large documents (e.g., 800-page reports) cannot fit into a single prompt. [\[Introducin...Generation\]](https://anthropic.skilljar.com/claude-with-the-anthropic-api/287763)
* Including everything leads to:
  * Prompt size limits
  * Higher cost
  * Slower performance
  * Reduced model effectiveness [\[Introducin...Generation\]](https://anthropic.skilljar.com/claude-with-the-anthropic-api/287763)

## Solution Approaches

### ❌ Option 1: Prompt Stuffing

* Put all text in the prompt
* Issues:
  * Size limits
  * High cost
  * Slow response
  * Less accurate processing [\[Introducin...Generation\]](https://anthropic.skilljar.com/claude-with-the-anthropic-api/287763)

### ✅ Option 2: RAG (Chunk + Retrieve)

* Preprocess: split document into chunks
* At query time:
  * Search relevant chunks
  * Include only those in prompt [\[Introducin...Generation\]](https://anthropic.skilljar.com/claude-with-the-anthropic-api/287763)

## Benefits of RAG

* Focuses model on relevant content
* Scales to **very large datasets**
* Works with **multiple documents**
* Reduces cost and latency [\[Introducin...Generation\]](https://anthropic.skilljar.com/claude-with-the-anthropic-api/287763)

## Challenges

* Requires preprocessing step
* Needs a **retrieval/search system**
* Risk of missing context
* Chunking strategy varies (size vs structure) [\[Introducin...Generation\]](https://anthropic.skilljar.com/claude-with-the-anthropic-api/287763)

## When to Use

* Large documents
* Multiple document sources
* Need for performance + cost optimization
* Trade-off: **complexity vs scalability** [\[Introducin...Generation\]](https://anthropic.skilljar.com/claude-with-the-anthropic-api/287763)

# ❓ 2. Q\&A

### Q1: What problem does RAG solve?

A: It handles large documents that exceed prompt limits by retrieving only relevant parts.

### Q2: Why is including all content in a prompt not ideal?

A: It hits size limits, increases cost, slows processing, and reduces effectiveness.

### Q3: What is the core idea of RAG?

A: Split documents into chunks and retrieve relevant chunks dynamically.

### Q4: What happens during preprocessing in RAG?

A: Documents are divided into smaller chunks.

### Q5: What is required at query time?

A: A search mechanism to find relevant chunks.

### Q6: What are common chunking strategies?

A: Equal-sized chunks or structure-based chunks (headers/sections).

### Q7: What trade-off does RAG introduce?

A: More implementation complexity for better scalability and efficiency.

# 🔑 4. Key Takeaways

* RAG = **Chunk + Retrieve + Generate**
* Avoids prompt size limitations
* Improves:
  * ✅ Accuracy (focused context)
  * ✅ Cost efficiency
  * ✅ Performance
* Requires:
  * Preprocessing
  * Retrieval mechanism
* Trade-off:
  * ✅ Scalable system
  * ❌ More engineering complexity

👉 For your Solution Architect perspective:

* RAG is essential for:
  * Enterprise knowledge systems
  * Document QA (insurance, finance, legal)
  * Internal copilots (SharePoint, PDFs, etc.)
