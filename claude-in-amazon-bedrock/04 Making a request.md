# Key Concepts

## 1. Making Your First AWS Bedrock Request

To call Claude through Amazon Bedrock, you need **three essential components**:

1. **Bedrock Runtime Client**
   * Connects your application to the Bedrock service.
2. **Model ID**
   * Identifies which Claude model you want to use.
3. **User Message**
   * The prompt or content sent to the model. [\[Making a request\]](https://anthropic-partners.skilljar.com/claude-in-amazon-bedrock/276719)

Example:

```python
import boto3

client = boto3.client(
    "bedrock-runtime",
    region_name="us-west-2"
)
```

 [\[Making a request\]](https://anthropic-partners.skilljar.com/claude-in-amazon-bedrock/276719)

***

## 2. Understanding Model IDs and Regions

### Problem

Claude models are **not available in every AWS region**.

Example:

* Claude model exists in `us-west-2`
* Your request comes from `us-east-1`
* Result: request fails with a model-not-found error. [\[Making a request\]](https://anthropic-partners.skilljar.com/claude-in-amazon-bedrock/276719)

### Key Takeaway

Before using a model ID, verify regional availability. [\[Making a request\]](https://anthropic-partners.skilljar.com/claude-in-amazon-bedrock/276719)

***

## 3. Inference Profiles

### What They Do

Inference profiles solve regional availability issues.

Instead of manually tracking:

* Which model exists in which region
* Which endpoint to call

AWS automatically routes requests to a region that hosts the model. [\[Making a request\]](https://anthropic-partners.skilljar.com/claude-in-amazon-bedrock/276719)

### Benefits

* Simplifies deployment
* Supports cross-region inference
* Reduces model availability errors
* Improves portability across AWS environments [\[Making a request\]](https://anthropic-partners.skilljar.com/claude-in-amazon-bedrock/276719)

### Where to Find Them

In AWS Bedrock Console:

```
Cross-region inference
```

Use the **Inference Profile ID**, not the model catalog ID. [\[Making a request\]](https://anthropic-partners.skilljar.com/claude-in-amazon-bedrock/276719)

***

## 4. User Message Structure

A user message has a standardized format:

```python
user_message = {
    "role": "user",
    "content": [
        {
            "text": "What's 1+1?"
        }
    ]
}
```

 [\[Making a request\]](https://anthropic-partners.skilljar.com/claude-in-amazon-bedrock/276719)

### Why Is Content a List?

Because messages can contain multiple content types:

* Text
* Images
* Other media types

This enables **multimodal interactions**. [\[Making a request\]](https://anthropic-partners.skilljar.com/claude-in-amazon-bedrock/276719)

***

## 5. Making the API Request

Use the Bedrock `converse()` API:

```python
response = client.converse(
    modelId=model_id,
    messages=[user_message]
)
```

 [\[Making a request\]](https://anthropic-partners.skilljar.com/claude-in-amazon-bedrock/276719)

***

## 6. Extracting the Response Text

The response contains metadata and generated content.

To retrieve the generated answer:

```python
response["output"]["message"]["content"][0]["text"]
```

 [\[Making a request\]](https://anthropic-partners.skilljar.com/claude-in-amazon-bedrock/276719)

***

## 7. Message Types

### User Messages

Input provided to the model.

```json
{
  "role": "user"
}
```

### Assistant Messages

Output generated by Claude.

```json
{
  "role": "assistant"
}
```

Both share the same structure:

```json
{
  "role": "...",
  "content": [...]
}
```

This consistency makes multi-turn conversations easier to build. [\[Making a request\]](https://anthropic-partners.skilljar.com/claude-in-amazon-bedrock/276719)

***

## 8. Multi-turn Conversation Design

Since both message types share the same schema:

```text
User
 ↓
Assistant
 ↓
User
 ↓
Assistant
```

Applications can continuously append messages to maintain context and conversation history. [\[Making a request\]](https://anthropic-partners.skilljar.com/claude-in-amazon-bedrock/276719)

# Solution Architect Notes

### Best Practices

✅ Use Inference Profiles instead of hardcoding model regions.

✅ Treat `content` as a multimodal container.

✅ Keep conversation history structured with alternating user/assistant messages.

✅ Build utility functions to extract output text from Bedrock responses.

✅ Handle region and model availability errors gracefully.*(Architectural recommendation based on the content.)*
