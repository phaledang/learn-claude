## 1. Summary

**Page:** *Defining resources*  
**URL:** <https://anthropic.skilljar.com/introduction-to-model-context-protocol/296699>

### Core concept

* **Resources (MCP)** = mechanism to expose **read-only data** from MCP server to client
* Similar to:
  * HTTP **GET endpoints**
  * Used for **fetching data**, NOT executing actions (tools handle actions)

### Key characteristics

* Request-response pattern:
  * Client → `ReadResourceRequest (URI)`
  * Server → `ReadResourceResult (data)` [\[Defining resources\]](https://anthropic.skilljar.com/introduction-to-model-context-protocol/296699)
* Data is fetched directly from server → injected into Claude prompt
* Avoids tool calls → more efficient

### Example use case

**Document mention feature (`@document_name`)**

* Operations:
  * List documents → for autocomplete
  * Fetch document content → inject into prompt [\[Defining resources\]](https://anthropic.skilljar.com/introduction-to-model-context-protocol/296699)
* Result:
  * Claude receives document content directly
  * No need to call tools

## 2. Types of Resources

### 2.1 Direct Resources

* Static URI
* No parameters
* Use for fixed datasets

```python
@mcp.resource("docs://documents", mime_type="application/json")
def list_docs() -> listreturn list(docs.keys())
```

***

### 2.2 Templated Resources

* Dynamic URI with parameters
* Parameters auto-parsed by SDK

```python
@mcp.resource("docs://documents/{doc_id}", mime_type="text/plain")
def fetch_doc(doc_id: str) -> str:
    if doc_id not in docs:
        raise ValueError(f"Doc with id {doc_id} not found")
    return docs[doc_id]
```

## 3. Data Handling

### Supported return types

* String
* JSON
* Binary (PDF, etc.) [\[Defining resources\]](https://anthropic.skilljar.com/introduction-to-model-context-protocol/296699)

### MIME types

* `application/json`
* `text/plain`
* `application/pdf`

### Behavior

* SDK auto-serializes return values
* No need for manual JSON conversion [\[Defining resources\]](https://anthropic.skilljar.com/introduction-to-model-context-protocol/296699)

***

## 4. Resource Execution Flow

```
Client request
   ↓
MCP Client
   ↓
MCP Server (resolve URI → call function)
   ↓
Return data → to client
   ↓
Injected into Claude prompt
```

## 6. Q\&A

### Q1. What problem do resources solve?

Provide structured way to fetch data → avoid tool calls for read-only data.

### Q2. Difference: resource vs tool?

| Aspect     | Resource          | Tool            |
| ---------- | ----------------- | --------------- |
| Purpose    | Read data         | Perform actions |
| Invocation | Client-driven     | Claude-driven   |
| Example    | Fetch doc content | Write file      |

***

### Q3. Why use resources instead of tools?

* Lower latency
* No reasoning step required
* Direct injection into prompt

***

### Q4. What identifies a resource?

A **URI string** (e.g., `docs://documents/{doc_id}`)

***

### Q5. How parameters work?

Parsed automatically from URI → passed to function

***

## 8. Key Takeaways

* Resources = **read-only data interface in MCP**
* Replace:
  * repeated prompt injection
  * unnecessary tool calls
* Key design:
  * URI-based access
  * Auto parameter parsing
* Best fit:
  * document systems
  * knowledge retrieval
  * structured prompt enrichment
* Complement tools → not replacement
