### MCP – Core Concepts (from current page)

#### 1. Three Server Primitives (core abstraction)

| Primitive     | Control Owner             | Responsibility                       | Typical Usage                                 |
| ------------- | ------------------------- | ------------------------------------ | --------------------------------------------- |
| **Tools**     | Model-controlled (Claude) | Execute capabilities autonomously    | External APIs, computation, actions           |
| **Resources** | Application-controlled    | Provide data/context to UI or prompt | Documents, autocomplete data, contextual info |
| **Prompts**   | User-controlled           | Trigger predefined workflows         | Buttons, slash commands, templates            |

***

### 2. Detailed Breakdown

#### **Tools (Model-controlled)**

* Fully controlled by the **AI model**
* Claude decides:
  * *when* to call a tool
  * *which* tool to use
* Results are consumed directly by Claude

**Purpose**

* Extend Claude’s capabilities beyond its base model

**Example**

* Request:*“calculate square root of 3”*
* Claude:
  * selects a **JavaScript execution tool**
  * runs the computation automatically

**Key property**

* Autonomous execution by model
* No UI or user trigger required [\[MCP review\]](https://anthropic.skilljar.com/introduction-to-model-context-protocol/296691)

***

#### **Resources (App-controlled)**

* Controlled by **application logic**
* App decides:
  * when to fetch data
  * what data to include

**Purpose**

* Provide **context or UI data**

**Usage patterns**

* Populate UI elements (e.g., autocomplete options)
* Inject additional context into prompts

**Example**

* “Add from Google Drive”:
  * App selects documents
  * App inserts content into context

**Key property**

* Used for **data retrieval and augmentation**, not execution [\[MCP review\]](https://anthropic.skilljar.com/introduction-to-model-context-protocol/296691)

***

#### **Prompts (User-controlled)**

* Triggered explicitly by **user actions**

**Trigger mechanisms**

* Buttons
* Slash commands
* Menu selections

**Purpose**

* Provide **predefined workflows**

**Example**

* Workflow buttons in Claude UI:
  * “Summarize”
  * “Rewrite”
  * etc.

**Key property**

* Deterministic, reusable workflows initiated by users [\[MCP review\]](https://anthropic.skilljar.com/introduction-to-model-context-protocol/296691)

***

### 3. Decision Guide (Important)

| Requirement                         | Use           |
| ----------------------------------- | ------------- |
| Add new capabilities for Claude     | **Tools**     |
| Provide data/context to system/UI   | **Resources** |
| Provide reusable workflows to users | **Prompts**   |

***

### 5. Key Insight (Critical for Design)

* Each primitive maps to a **different control layer**:
  * Tools → **Model layer**
  * Resources → **Application layer**
  * Prompts → **User layer**

➡️ MCP separates concerns cleanly across:

* autonomy (model)
* data/context (app)
* interaction (user)

***

### 6. Practical Implication (Architecture decisions)

| Scenario                        | Correct Primitive |
| ------------------------------- | ----------------- |
| Call API / DB / GitHub          | Tools             |
| Inject document into prompt     | Resources         |
| Create “Generate Report” button | Prompts           |

***

### 7. Common Mistake

* Using **tools** for data retrieval → should be **resources**
* Using **prompts** for automation → should be **tools**
* Mixing control boundaries → breaks MCP design principles

***

### 8. Summary (1-line each)

* **Tools** = "Let Claude act"
* **Resources** = "Let app provide data"
* **Prompts** = "Let users trigger workflows"
