## 1. Summary

**Page:** [Implementing a client](https://anthropic.skilljar.com/introduction-to-model-context-protocol/296696)

### Key concepts

* MCP architecture separates:
  * **Client** → application-side integration
  * **Server** → tool/resource provider
* In real projects:
  * Usually implement **either client OR server**
  * This course demonstrates **both for learning purpose**

### Client structure

* **MCP Client (custom wrapper)**
  * Encapsulates logic
  * Handles cleanup (resource management)
* **Client Session (SDK object)**
  * Actual connection to MCP server
  * Requires proper lifecycle handling

### Responsibilities of MCP Client

* Enable application to:
  * Discover tools
  * Execute tools
* Acts as a **bridge between application logic and MCP server**

### Integration points in app flow

* CLI (or app code) uses client to:
  1. Get available tools → send to Claude
  2. Execute tools → when Claude requests

### Core functions

#### list\_tools()

```python
async def list_tools(self) -> list[types.Tool]:
    result = await self.session().list_tools()
    return result.tools
```

* Calls MCP server
* Returns tool definitions (metadata + schema)

#### call\_tool()

```python
async def call_tool(
    self,
    tool_name: str,
    tool_input: dict
) -> types.CallToolResult | None:
    return await self.session().call_tool(tool_name, tool_input)
```

* Executes tool with:
  * `tool_name`
  * `tool_input`
* Returns execution result

### Testing

* Run client:

```bash
uv run mcp_client.py
```

* Expected:
  * Connect to server
  * Print available tools

### End-to-end flow

1. App asks client → `list_tools()`
2. Tools sent to Claude
3. Claude chooses tool
4. App calls → `call_tool()`
5. Result returned → Claude generates final response

## 2. Q\&A

**Q1. What is the role of the MCP client?**

* Acts as the communication layer between application code and MCP server.

**Q2. What are the two main components of the client?**

* MCP Client (custom wrapper)
* Client Session (SDK connection)

**Q3. Why wrap the client session in a custom class?**

* To handle resource management and cleanup automatically.

**Q4. What does `list_tools()` do?**

* Retrieves all available tools from the MCP server.

**Q5. What does `call_tool()` do?**

* Executes a specific tool with inputs.

**Q6. When does the application call `call_tool()`?**

* When Claude decides a tool should be used.

**Q7. What is the typical runtime flow with Claude?**

* Send tools → Claude selects → execute tool → return result → Claude responds.

## 4. Key Takeaways

* MCP client = **execution bridge**
* Core operations:
  * `list_tools()` → discovery
  * `call_tool()` → execution
* Claude **does not execute tools directly**
  * Application must call MCP via client
* Proper abstraction:
  * Wrap session → safe lifecycle handling
* Flow pattern:

```text
App → Client → MCP Server → Result → Claude → Response
```
