## 1. Summary — MCP Clients

**Source:** <https://anthropic.skilljar.com/introduction-to-model-context-protocol/296690>

### Core concept

* **MCP Client = communication bridge**
  * Connects **your app/server ↔ MCP server**
  * Provides access to **tools exposed by MCP server**
  * Handles:
    * message exchange
    * protocol complexity
* Application does **not need to manage low-level communication** [\[MCP clients\]](https://anthropic.skilljar.com/introduction-to-model-context-protocol/296690)

### Key capability — Transport agnostic

* Client–server communication supports multiple protocols:
  * **STDIO (most common, local dev)**
  * **HTTP**
  * **WebSockets**
  * Other network protocols [\[MCP clients\]](https://anthropic.skilljar.com/introduction-to-model-context-protocol/296690)

### MCP Message Types

| Category                    | Purpose                          |                                                             |
| --------------------------- | -------------------------------- | ----------------------------------------------------------- |
| `ListToolsRequest / Result` | Discover available tools         |                                                             |
| `CallToolRequest / Result`  | Execute a tool and return output |  [\[MCP clients\]](https://anthropic.skilljar.com/introduction-to-model-context-protocol/296690) |

***

### End-to-end execution flow (important for architecture)

**User query → tools → execution → response**

1. User sends query to application server
2. Server needs tool list → asks MCP client
3. MCP client → MCP server
   * sends `ListToolsRequest`
   * receives `ListToolsResult`
4. Server sends prompt + tools → Claude
5. Claude decides to use a tool
6. Server asks MCP client to execute tool
7. MCP client:
   * sends `CallToolRequest`
   * MCP server executes (e.g., GitHub API)
8. Result flows back
9. Server sends result → Claude
10. Claude generates final answer
11. Response returned to user

→ **MCP client abstracts all communication complexity** [\[MCP clients\]](https://anthropic.skilljar.com/introduction-to-model-context-protocol/296690)

## 2. Q\&A

**Q1. What is the main role of an MCP client?**

* Acts as a communication layer between application/server and MCP server

**Q2. Why is MCP called transport agnostic?**

* Because it supports multiple communication protocols (STDIO, HTTP, WebSockets)

**Q3. What is `ListToolsRequest` used for?**

* To retrieve available tools from MCP server

**Q4. What is `CallToolRequest` used for?**

* To execute a tool with input arguments

**Q5. Why use MCP client instead of direct API integration?**

* It abstracts protocol handling and standardizes tool access

**Q6. What is the typical local development setup?**

* Client and server communicate via standard input/output

**Q7. Where does Claude fit in the flow?**

* Claude decides whether to use tools and generates the final answer

## 4. Key Takeaways

* MCP Client = **standardized integration layer**
* Separates:
  * **app logic**
  * **tool access**
  * **protocol handling**
* Supports **multiple communication transports**
* Enables **dynamic tool discovery + execution**
* Works with Claude to:
  * decide tool usage
  * produce final responses
* Core for building:
  * **agent systems**
  * **tool-enabled LLM apps**
