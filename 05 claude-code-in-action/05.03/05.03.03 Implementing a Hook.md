## 1) Summary (page: <https://anthropic.skilljar.com/claude-code-in-action/312003>)

**Topic: Implementing a Hook in Claude Code**

* Demonstrates **PreToolUse hook** to intercept tool execution before it runs.
* Example use case: **prevent access to sensitive files (.env)** during a session.
* Key idea:
  * Hooks act as a **guard layer** before tools (Read, Grep, Bash, etc.) execute.
  * They inspect input and decide **allow vs block**.

### Core flow

1. Claude attempts to call a tool (e.g., `Read`)
2. Hook intercepts the request
3. Hook logic checks arguments
4. If violation → block execution via exit code

### Configuration

Add hook in `.claude/settings.local.json`:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Read",
        "hooks": [
          {
            "type": "command",
            "command": "node $PWD/hooks/read_hook.js"
          }
        ]
      }
    ]
  }
}
```

### Hook implementation

`hooks/read_hook.js`:

```javascript
process.stdin.setEncoding("utf8");

let input = "";

process.stdin.on("data", (d) => (input += d));

process.stdin.on("end", () => {
  const toolArgs = JSON.parse(input);
  const readPath = toolArgs.tool_input?.file_path || "";

  if (readPath.includes(".env")) {
    console.error("You cannot read the .env file");
    process.exit(2); // block
  }

  process.exit(0); // allow
});
```

### Important mechanics

* Input comes via **stdin (JSON payload)**
* Output:
  * `stderr` → message shown to Claude
  * exit code `2` → block execution
  * exit code `0` → allow execution
* Only applies to **Read tool**
* Other tools require separate logic due to different input structures

### Tool input differences

| Tool | Input structure     |
| ---- | ------------------- |
| Read | `{ file_path }`     |
| Grep | `{ pattern, path }` |
| Bash | `{ command }`       |

### Security recommendation

* Combine hooks with:

```text
permissions.deny: "Read(**/.env)"
```

* Hooks = dynamic control
* permissions.deny = consistent enforcement across tools [\[Implementing a hook\]](https://anthropic.skilljar.com/claude-code-in-action/312003)

## 2) Q\&A

**Q1. What is a PreToolUse hook?**  
A mechanism that intercepts tool calls before execution and allows validation or blocking.

**Q2. How does the hook block execution?**  
By returning a non-zero exit code (e.g., `process.exit(2)`).

**Q3. Where does the hook receive input from?**  
Via `stdin` as a JSON payload representing the tool call.

**Q4. How is the error message returned to Claude?**  
Anything written to `stderr` is displayed to Claude.

**Q5. Why does the example only block `.env` for Read tool?**  
Different tools have different input shapes, requiring separate handling.

**Q6. Why is permissions.deny recommended?**  
It provides uniform protection across all tools instead of tool-specific checks.

## 4) Key Takeaways

* Hooks = **policy enforcement at tool execution boundary**
* PreToolUse = **intercept → inspect → allow/block**
* Input handling:
  * JSON via stdin
  * tool-specific schema
* Blocking mechanism:
  * exit code != 0
  * message via stderr
* **Security pattern**
  * Hook → dynamic checks
  * permissions.deny → static enforcement
* Must implement **per-tool validation logic** for full protection
