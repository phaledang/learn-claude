## 2. Key Concepts

### Overview

* **JSON message types (MCP - Model Context Protocol)**
  * Standardized message schema used for communication between **client ↔ server ↔ tools**.
  * Defines how requests, responses, notifications, and errors are structured.

### Core Message Types

#### 1. Request Message

* Sent from **client → server**
* Purpose: invoke an operation (e.g., call tool, initialize session)
* Key fields:
  * `id` → unique identifier for request-response correlation
  * `method` → action name
  * `params` → input arguments

```json
{
  "id": "123",
  "method": "tool.call",
  "params": {
    "name": "research",
    "arguments": { "topic": "AI trends" }
  }
}
```

***

#### 2. Response Message

* Sent from **server → client**
* Purpose: return result of a request
* Key fields:
  * `id` → must match request id
  * `result` → successful output
  * `error` → error object (if failed)

```json
{
  "id": "123",
  "result": {
    "summary": "AI trends report..."
  }
}
```

***

#### 3. Notification Message

* Sent **without expecting response**
* Purpose:
  * log messages
  * progress updates
  * state changes
* No `id` field

```json
{
  "method": "progress",
  "params": {
    "current": 50,
    "total": 100
  }
}
```

***

#### 4. Error Object

* Returned inside response when failure occurs
* Structure:
  * `code`
  * `message`
  * `data` (optional)

```json
{
  "id": "123",
  "error": {
    "code": -32603,
    "message": "Internal error"
  }
}
```

***

### Message Flow

* **Typical lifecycle**
  1. Client sends **Request**
  2. Server processes
  3. Server emits **Notifications** (optional) during execution
  4. Server returns **Response**

```text
Client → Request → Server  
Server → Notification(s) → Client  
Server → Response → Client  
```

***

### Design Principles

* **Request–Response pairing**
  * Every request must have a matching response (`id` correlation)

* **Asynchronous-friendly**
  * Notifications allow non-blocking updates

* **Loose coupling**
  * Client decides how to handle responses and notifications

* **Extensibility**
  * New methods can be introduced without breaking protocol

***

### Relationship with MCP Execution

* Request → triggers tool execution
* Notification → real-time signals (logs, progress)
* Response → final result returned

## 3. Q\&A

**Q1. What is the role of JSON message types in MCP?**

* Define standardized communication between client and server

**Q2. What distinguishes a notification from a response?**

* Notification has no `id` and does not expect a reply

**Q3. Why is `id` important in requests?**

* Used to match responses with requests

**Q4. Can a request return both result and error?**

* No, only one is present

**Q5. When are notifications used?**

* During long-running operations for progress/log updates
