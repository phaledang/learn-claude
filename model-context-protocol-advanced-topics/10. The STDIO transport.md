## 2. Key Concepts

### 2.1 Transport in MCP (Model Context Protocol)

* **Transport = communication channel** used to exchange JSON messages between MCP client and server [\[The STDIO transport\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296291)
* Possible implementations:
  * HTTP requests
  * WebSockets
  * stdio (standard input/output)
* Core idea:
  * MCP is transport-agnostic → same message format, different delivery mechanisms

***

### 2.2 STDIO Transport (Standard Input/Output)

#### Definition

* **stdio transport**:
<img width="1920" height="1080" alt="image" src="https://github.com/user-attachments/assets/c28860b4-d34c-412f-bd3d-16ec979e10f5" />

  * Client starts the MCP server as a **subprocess**
  * Communication happens via:
    * `stdin` → input stream
    * `stdout` → output stream [\[The STDIO transport\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296291)

#### How it works

<img width="1920" height="1080" alt="image" src="https://github.com/user-attachments/assets/809a2700-e7da-46be-a7de-2f40596a9b91" />

* Client → sends message → writes to `stdin`
* Server → sends response → writes to `stdout`
* Communication is:
  * **bidirectional**
  * **asynchronous (either side can initiate)** [\[The STDIO transport\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296291)

#### Key constraint

* Only works when:
  * Client and server are on **same machine** [\[The STDIO transport\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296291)

***

### 2.3 Communication Flow Model

<img width="1920" height="1080" alt="image" src="https://github.com/user-attachments/assets/e6d700c6-36e9-4e63-afb6-cb2b486a795a" />

#### Four communication scenarios

1. Client → Server (request) → via stdin
2. Server → Client (response) → via stdout
3. Server → Client (request) → via stdout
4. Client → Server (response) → via stdin [\[The STDIO transport\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296291)

➡️ Insight:

* **Both sides can initiate communication at any time**
* Enables **full duplex interaction**

***

### 2.4 Message Types

<img width="1920" height="1080" alt="image" src="https://github.com/user-attachments/assets/e774373b-5c07-4f45-b579-9ec8929090a9" />

* MCP defines two categories:
  * **Requests → must have responses**
  * **Notifications → no response required** [\[The STDIO transport\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296291)

➡️ Design implication:

* System must:
  * track request/response pairs
  * handle fire-and-forget notifications

***

### 2.5 MCP Connection Handshake (Critical)

Every connection must follow this **3-step initialization sequence**:

1. **Initialize Request** (Client → Server)
2. **Initialize Result** (Server → Client)
3. **Initialized Notification** (Client → Server, no response) [\[The STDIO transport\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296291)

➡️ Constraint:

* No other operations (tool calls, prompts, etc.) are allowed **before handshake completes**

***

### 2.6 Development Workflow (STDIO)

#### Testing server directly

* Run server:
  * `uv run server.py`
* Server:
  * listens on `stdin`
  * outputs via `stdout`
* Allows:
  * manual JSON input in terminal
  * real-time response observation [\[The STDIO transport\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296291)

➡️ Use case:

* Debugging MCP servers **without building a client**

***

### 2.7 Why STDIO Matters

#### Advantages

* Simple to implement
* Full bidirectional communication
* Ideal for:
  * development
  * testing
  * local debugging [\[The STDIO transport\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296291)

#### Limitations

* Not suitable for:
  * distributed systems
  * remote communication
* Requires:
  * same machine execution [\[The STDIO transport\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296291)

***

### 2.8 Conceptual Role in MCP Design

* STDIO represents the**"ideal baseline" transport**:
  * No restrictions on who can initiate communication
* Used to:
  * understand full MCP capabilities
  * compare with constrained transports (e.g., HTTP) [\[The STDIO transport\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296291)

## 3. Q\&A

**Q1. What is a transport in MCP?**  
A communication mechanism used to transmit JSON messages between client and server. [\[The STDIO transport\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296291)

**Q2. How does stdio transport work?**  
Client launches server as subprocess and communicates via stdin (input) and stdout (output). [\[The STDIO transport\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296291)

**Q3. Can both client and server initiate messages?**  
Yes, both sides can send requests at any time. [\[The STDIO transport\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296291)

**Q4. Why is stdio limited?**  
Because it only works when both processes run on the same machine. [\[The STDIO transport\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296291)

**Q5. What must happen before any MCP operation?**  
A 3-step handshake: Initialize → Result → Initialized notification. [\[The STDIO transport\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296291)

**Q6. What are the two message types?**

* Requests (require responses)
* Notifications (no response) [\[The STDIO transport\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296291)
