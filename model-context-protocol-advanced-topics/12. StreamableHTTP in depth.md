## 2. Key concepts

### StreamableHTTP overview

* **StreamableHTTP** = a transport mechanism in **Model Context Protocol (MCP)** that enables bidirectional-like communication over HTTP.
* Designed to solve a core limitation:
  * HTTP is **client → server only**
  * But MCP requires **server → client communication** (e.g., sampling, notifications, logging). [\[Streamable...P in depth\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296286)

***

### Core problem

* Some MCP capabilities depend on **server-initiated messages**:
  * Sampling requests
  * Notifications
  * Logging streams
* Standard HTTP does **not support server push natively**. [\[Streamable...P in depth\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296286)

***

### Solution: Server-Sent Events (SSE)

* StreamableHTTP uses **Server-Sent Events (SSE)** to work around HTTP limitations.
* SSE enables:
  * A **long-lived HTTP response**
  * Continuous **server → client streaming**
* This creates an **asynchronous communication channel**. [\[Streamable...P in depth\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296286)

***

### Connection lifecycle

#### Initialization phase

* Client → Server: `Initialize Request`
* Server → Client: `Initialize Result`
  * Contains **mcp-session-id** (critical)
* Client → Server: `Initialized Notification` (includes session ID)

**Key point:**

* `mcp-session-id`
  * Uniquely identifies the client
  * Must be included in **all subsequent requests** [\[Streamable...P in depth\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296286)

***

### Persistent SSE connection

* Client establishes a **GET request** → opens SSE stream
* Server uses this to:
  * Send notifications
  * Send requests
  * Push messages anytime

✅ This is the **core workaround enabling server → client communication** [\[Streamable...P in depth\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296286)

***

### Dual SSE connection model

#### 1. Primary SSE connection

* Long-lived
* Handles:
  * Server-initiated requests
  * Progress notifications
* Stays open continuously

#### 2. Tool-specific SSE connection

* Created **per tool call**
* Handles:
  * Tool execution results
  * Related logging
* Automatically closes after completion

***

### Message routing

* Different message types go through different channels:

| Message Type           | Connection Type   |                                                             |
| ---------------------- | ----------------- | ----------------------------------------------------------- |
| Progress notifications | Primary SSE       |                                                             |
| Logging messages       | Tool-specific SSE |                                                             |
| Tool results           | Tool-specific SSE |  [\[Streamable...P in depth\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296286) |

***

### Configuration flags (advanced behavior)

Two flags can disable the workaround:

* `stateless_http`
* `json_response`

If set to `true`:

* SSE workaround **may break**
* Server → client communication becomes limited

✅ Use cases:

* Simpler HTTP scenarios
* Stateless environments

⚠️ Trade-off:

* Lose full MCP capabilities (sampling, streaming, etc.) [\[Streamable...P in depth\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296286)

***

### Complexity vs other transports

* StreamableHTTP is **more complex than STDIO transport** because:
  * Needs session tracking
  * Uses persistent streaming
  * Manages multiple connections

***

### Key architectural insights

* Requires **stateful session management** (via session ID)
* Uses **SSE as a pseudo bidirectional layer**
* Implements **multi-channel communication strategy**
* Automatically manages connection lifecycle per tool

## 3. Q\&A

**Q1: Why does StreamableHTTP exist?**

* To enable server-initiated communication over HTTP, which normally only supports client-initiated requests. [\[Streamable...P in depth\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296286)

**Q2: What enables server → client messaging?**

* Server-Sent Events (SSE), which maintain a long-lived streaming connection. [\[Streamable...P in depth\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296286)

**Q3: What is the role of `mcp-session-id`?**

* Identifies the client session and must be included in all subsequent requests. [\[Streamable...P in depth\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296286)

**Q4: Why are there two SSE connections?**

* To separate:
  * Persistent communication (primary)
  * Tool execution lifecycle (tool-specific) [\[Streamable...P in depth\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296286)

**Q5: What happens if `stateless_http=true`?**

* The SSE workaround may not function, limiting MCP features. [\[Streamable...P in depth\]](https://anthropic.skilljar.com/model-context-protocol-advanced-topics/296286)
